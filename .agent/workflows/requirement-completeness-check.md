# 需求调研完整性检测

> **author**: AI助手  
> **date**: 2026-03-31  
> **version**: v1.0  
> **description**: 需求调研完整性和质量检查清单，用于确保需求调研覆盖全面、信息准确、可落地

---

## 📋 概述

本检查清单用于评估需求调研的完整性和质量，涵盖调研前准备、调研过程、调研报告整理三个阶段。适用于B端系统（特别是ERP、供应链系统）的需求调研工作，确保需求分析全面、无遗漏，为PRD编写和系统设计提供可靠输入。

## 🔍 调研前准备检查

### 调研计划完整性
| 检查项 | 检查方法 | 通过标准 | 权重 |
|--------|----------|----------|------|
| **调研主题明确** | 检查调研目标是否清晰 | 有明确的调研主题，如"采购流程优化"、"库存盘点流程改进" | 高 |
| **调研对象完整** | 检查涉及角色是否全覆盖 | 包含主要用户角色、次要用户角色、管理者、上下游协作者 | 高 |
| **调研方式合理** | 检查调研方法是否恰当 | 根据场景选择访谈、问卷、现场观察、数据分析等合适方法 | 中 |
| **时间安排合理** | 检查调研时间是否充足 | 有明确的调研时间表，预留准备和整理时间 | 中 |
| **背景资料准备** | 检查是否了解业务背景 | 已阅读相关业务文档（公司介绍、业务流程等） | 高 |

### 问题清单质量
| 检查项 | 检查方法 | 通过标准 | 权重 |
|--------|----------|----------|------|
| **6个维度覆盖** | 检查问题是否涵盖6个维度 | 业务背景、现状痛点、期望目标、业务场景、数据与规则、权限与协同 | 高 |
| **问题针对性** | 检查问题是否针对调研主题 | 问题与调研主题高度相关，避免泛泛而问 | 高 |
| **开放性问题比例** | 检查问题类型是否合理 | 开放性问题≥60%，引导用户详细描述 | 中 |
| **追问点准备** | 检查是否有追问准备 | 关键问题有预设的追问点，深入挖掘 | 中 |
| **问题逻辑顺序** | 检查问题排列是否合理 | 从宏观到微观，从背景到细节 | 低 |

## 📊 检查结果评分

### 评分标准
- **高权重项**：每项不合格扣5分
- **中权重项**：每项不合格扣3分  
- **低权重项**：每项不合格扣1分
- **总分**：100分

### 质量等级
| 分数范围 | 等级 | 说明 | 行动建议 |
|----------|------|------|----------|
| 90-100 | 优秀 | 调研全面深入，报告结构完整，需求质量高 | 可直接用于PRD编写，建议仅做微小优化 |
| 75-89 | 良好 | 调研基本全面，报告结构完整，需求质量良好 | 建议补充缺失的高权重项，优化中低权重项 |
| 60-74 | 及格 | 调研有主要遗漏，报告结构基本完整 | 必须补充高权重缺失项，建议重新调研关键部分 |
| <60 | 不合格 | 调研严重不完整，需求质量差 | 需要重新规划并执行调研 |

## 🚀 最佳实践建议

### 调研执行建议
1. **组合使用多种方法**: 访谈+问卷+现场观察+数据分析
2. **分层调研**: 先管理层（战略层），再执行层（操作层）
3. **现场观察**: 重要流程尽量现场观察，发现隐性问题
4. **数据验证**: 用户描述与实际数据对比验证
5. **跨部门验证**: 关键流程跨部门交叉验证

### 报告编写建议
1. **尽早编写**: 调研后24小时内完成初稿
2. **可视化优先**: 多用图表（流程图、矩阵、优先级图）
3. **用户验证**: 报告初稿请关键用户Review
4. **版本管理**: 报告有版本号，重要修改有记录
5. **关联PRD**: 报告中的需求直接关联到PRD需求项

### 需求分析建议
1. **需求分类**: 按功能、非功能、业务规则、数据等分类
2. **需求优先级**: 使用MoSCoW法（Must have, Should have, Could have, Won't have）
3. **需求溯源**: 每个需求可追溯到调研中的原始输入
4. **冲突解决**: 识别需求冲突并给出解决方案
5. **假设明确**: 明确列出所有假设，避免隐含假设
