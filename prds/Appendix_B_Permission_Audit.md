## B2. 权限模型说明（RBAC/ABAC）

### 权限模型：RBAC + 数据范围控制

AI Factory Creator采用**基于角色的访问控制（RBAC）** 为主，结合**基于属性的访问控制（ABAC）** 进行数据范围控制。

#### 权限粒度
1. **菜单级权限**：控制用户可见的菜单和功能模块
2. **功能级权限**：控制具体功能（创建、编辑、删除、导出等）的可用性
3. **按钮级权限**：控制界面按钮的显示和可用状态
4. **字段级权限**：控制表单字段的可见、可编辑、必填等状态
5. **数据范围权限**：控制用户可访问的数据范围（按工厂、车间、项目等）

#### 数据范围控制策略
- **按组织架构**：用户只能访问自己所在工厂/车间/部门的数据
- **按项目归属**：用户只能访问自己创建或参与的项目
- **按角色层级**：高层级角色可查看下级角色数据，反之不可
- **按数据分类**：敏感数据（如成本、工艺参数）需要额外权限

#### 权限继承规则
1. **角色继承**：高级角色继承低级角色的所有权限
2. **数据范围继承**：上级组织可查看下级组织数据
3. **项目权限继承**：项目负责人可管理项目所有数据和成员

## B4. 审计留痕规范（Audit Trail）

### 必须留痕的操作清单

| 操作类别 | 具体操作 | 留痕级别 | 触发条件 |
|----------|---------|---------|---------|
| **数据创建** | 创建项目、创建模型、创建实例、创建用户 | 详细 | 每次操作成功时 |
| **数据修改** | 修改项目属性、修改模型、移动实例、更新绑定 | 详细 | 每次修改保存时 |
| **数据删除** | 删除项目、删除模型、删除实例、删除用户 | 详细 | 删除操作执行时 |
| **状态变更** | 发布项目、启用/禁用模型、激活/停用实例 | 详细 | 状态变更时 |
| **权限变更** | 修改用户权限、变更角色、调整数据范围 | 详细 | 权限配置保存时 |
| **系统操作** | 登录/登出、导出数据、导入数据、系统配置 | 一般 | 操作执行时 |
| **数据访问** | 查看敏感数据（成本、工艺参数等） | 一般 | 访问敏感数据时 |
| **批量操作** | 批量导入、批量导出、批量修改 | 详细 | 批量操作完成时 |

### 留痕字段与前后值

| 审计字段 | 数据类型 | 是否必填 | 说明 |
|----------|---------|---------|------|
| audit_id | VARCHAR(36) | 是 | 审计记录唯一标识 |
| user_id | VARCHAR(36) | 是 | 操作用户ID |
| username | VARCHAR(50) | 是 | 操作用户名 |
| operation_type | VARCHAR(50) | 是 | 操作类型：CREATE, UPDATE, DELETE, LOGIN, EXPORT等 |
| operation_target | VARCHAR(100) | 是 | 操作目标：PROJECT, ASSET_MODEL, ASSET_INSTANCE等 |
| target_id | VARCHAR(36) | 否 | 目标对象ID（如项目ID、模型ID） |
| operation_time | DATETIME | 是 | 操作时间 |
| ip_address | VARCHAR(45) | 是 | 客户端IP地址 |
| user_agent | TEXT | 否 | 用户代理（浏览器/客户端信息） |
| operation_details | JSON | 是 | 操作详细信息 |
| before_value | JSON | 否 | 操作前数据（修改/删除操作时记录） |
| after_value | JSON | 否 | 操作后数据（创建/修改操作时记录） |
| change_summary | TEXT | 否 | 变更摘要（人工可读） |
| status | VARCHAR(20) | 是 | 操作状态：SUCCESS, FAILED, PARTIAL |
| error_message | TEXT | 否 | 失败时的错误信息 |
| request_id | VARCHAR(36) | 否 | 请求跟踪ID，用于关联日志 |

#### operation_details JSON结构示例
```json
{
  "operation": "UPDATE_PROJECT",
  "project_id": "proj_001",
  "changed_fields": ["project_name", "layout_config"],
  "source": "WEB_UI",
  "session_id": "session_123456"
}
```

#### before_value/after_value JSON结构示例
```json
{
  "project_name": {
    "before": "上海工厂布局V1",
    "after": "上海智能工厂布局V2"
  },
  "layout_config": {
    "before": {"width": 100, "height": 50},
    "after": {"width": 120, "height": 60}
  }

```

### 查询入口与导出规则

#### 审计日志查询入口
1. **系统管理后台** → **审计日志**：系统管理员专用查询界面
2. **项目详情页** → **操作历史**：查看特定项目的操作记录
3. **模型详情页** → **变更历史**：查看模型版本和修改记录
4. **用户详情页** → **操作记录**：查看特定用户的操作历史
5. **API接口**：`GET /api/v1/audit-logs` 支持程序化查询

#### 查询条件支持
- **时间范围**：按操作时间筛选（支持相对时间和绝对时间）
- **用户筛选**：按用户ID、用户名、角色筛选
- **操作类型**：按操作类型（CREATE/UPDATE/DELETE等）筛选
- **目标对象**：按操作目标（项目、模型、实例等）筛选
- **状态筛选**：按操作成功/失败状态筛选
- **关键词搜索**：在变更摘要、错误信息中搜索

#### 导出规则
1. **导出权限**：仅系统管理员可导出审计日志
2. **导出格式**：支持CSV、Excel、PDF格式
3. **数据范围**：支持按查询条件导出筛选后的数据
4. **导出限制**：单次最多导出10万条记录，超过需分批导出
5. **导出审批**：敏感数据（用户操作全量日志）导出需二级审批
6. **导出记录**：所有导出操作本身记录审计日志

### 保留期限

| 审计日志类型 | 在线保留期限 | 归档保留期限 | 删除策略 |
|-------------|------------|------------|---------|
| **操作审计日志** | 180天 | 5年 | 180天后自动归档，5年后自动删除 |
| **登录审计日志** | 90天 | 3年 | 90天后自动归档，3年后自动删除 |
| **数据变更日志** | 365天 | 永久 | 365天后自动归档，永久保留关键变更 |
| **权限变更日志** | 永久 | 永久 | 在线永久保留，定期备份 |
| **系统异常日志** | 30天 | 1年 | 30天后自动归档，1年后自动删除 |
| **性能监控日志** | 7天 | 90天 | 7天后自动归档，90天后自动删除 |

#### 保留策略说明
1. **在线保留**：数据存储在在线数据库中，支持快速查询和分析
2. **归档保留**：数据转移到低成本存储（如对象存储），查询速度较慢
3. **自动清理**：系统按保留策略自动清理过期数据，清理前发送通知
4. **合规要求**：保留期限符合公司信息安全政策和相关法律法规
5. **备份策略**：审计日志每日备份，备份保留期限为10年

#### 特殊情况处理
- **法律调查**：涉及法律调查的数据，保留期限延长至调查结束
- **安全事件**：涉及安全事件的数据，永久保留作为证据
- **用户请求**：用户可申请延长特定数据的保留期限（需审批）
- **系统迁移**：系统迁移时审计日志同步迁移，保留期限重新计算
